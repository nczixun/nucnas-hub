## 为什么要迁移NAS数据

随着存储需求增长或设备升级，数据迁移成为NAS用户必须面对的问题。正确的数据迁移可以确保业务连续性，避免数据丢失风险。

## 迁移前的准备工作

### 1. 评估现有数据

- 统计总数据量
- 识别大文件分布
- 评估文件夹结构
- 检查文件权限设置

### 2. 选择迁移方式

| 迁移方式 | 适用场景 | 速度 | 难度 |
|----------|----------|------|------|
| 网络传输 | 跨设备迁移 | 取决于网络 | 简单 |
| 外接硬盘 | 同规格设备 | 快 | 中等 |
| RAID迁移 | 硬盘升级 | 慢 | 复杂 |

### 3. 备份策略

**重要原则：迁移前务必完整备份**

推荐采用321备份原则：
- 3份副本
- 2种不同介质
- 1份异地存储

## 详细迁移步骤

### 第一步：备份元数据

```bash
# 导出Docker容器配置
docker-compose -f /path/to/compose.yml config > docker-compose-backup.yml

# 记录所有服务配置
cat > services-backup.txt <<EOF
Jellyfin: 192.168.1.100:8096
HomeAssistant: 192.168.1.100:8123
Portainer: 192.168.1.100:9000
EOF
```

### 第二步：传输数据

#### 方案A：rsync同步（推荐）

```bash
# 在目标NAS上执行
rsync -avz --progress \
  -e 'ssh -p 22' \
  /source/path/ \
  user@target-nas:/target/path/
```

#### 方案B：SMB共享传输

1. 源NAS开启SMB共享
2. 目标NAS挂载SMB
3. 使用拖拽方式复制

### 第三步：验证数据

#### 校验文件完整性

```bash
# 生成校验和
find /data -type f -exec md5sum {} \; > checksum.txt

# 验证校验和
md5sum -c checksum.txt
```

#### 检查Docker容器状态

1. 重新创建Docker网络
2. 逐一启动容器
3. 检查服务可用性

## 常见问题解决

### 问题1：权限丢失

**解决方案：**

```bash
# 使用rsync保留权限
rsync -avz --perms --owner --group ...
```

### 问题2：服务无法启动

**排查步骤：**

1. 检查网络配置是否一致
2. 确认端口映射是否冲突
3. 验证存储路径挂载

### 问题3：传输中断

**建议：**

- 使用screen/tmux保持传输
- 考虑使用rsync的--partial参数支持断点续传

## 迁移后的优化

### 1. 清理冗余文件

```bash
# 查找重复文件
fdupes -r /data > duplicates.txt

# 清理临时文件
find /data -type f -name "*.tmp" -delete
```

### 2. 重新索引

- 重新扫描媒体库
- 更新文件索引
- 验证Docker日志

## 时间规划建议

| 数据量 | 预估时间 | 推荐时段 |
|--------|----------|----------|
| < 1TB | 2-4小时 | 夜间 |
| 1-5TB | 6-12小时 | 周末 |
| > 5TB | 24小时+ | 长假 |

## 总结

NAS数据迁移是一项需要谨慎操作的任务。关键要点：

1. **充分备份**是前提
2. **选择合适的迁移方式**
3. **完整性验证**不可省略
4. **保留回滚方案**

按照本指南操作，可以最大程度确保数据安全平滑迁移。
